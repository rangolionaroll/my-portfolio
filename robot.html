Robot Simulation

A robot simulation in Unity to view human-robot interaction scenarios.

Role: Developer

This project is another project I completed as a contract developer. It is a simulation that shows how a robot would interact in real life as a quick prototype before actual implementation. The main idea that the clients wanted to test was how the robot would interact with humans and potentially respond in scenarios such as a human waving or getting too close to it.

Robot Logic

    The main part of the project that I worked on was the logic for the robot, mainly programming the movements and reactions that the robot was supposed to display. The base movements and animations include:

  • Start-up at a certain location.
  • Locate the nearest table.
  • Navigate to the table.
  • Face the table and wipe until clean.
  • Repeat this process for all tables.
  • Navigate back to a certain location.
  • Shut-Down.

    • Most of the wayfinding logic was completed using Unity’s NavMesh component to create a walkable surface area and then setting up waypoints for the places the robot was to navigate to. As some tables were longer than other tables, they all contained several waypoints for the robot to stop at and wipe before moving on to the next table. At each waypoint, the robot would shoot a raycast towards the table and get the normal of the surface so it could properly position itself to face the table.

    Next was the interaction that the robot would have with humans that were present in the environment. The robot has two sensors, one which has a shorter radius, but a larger distance and one that is the reverse. If someone was in the field-of-view of the first sensor, the robot would nod at them or wave if they waved. If they approached the robot and got close enough to the second field-of-view, the robot would communicate that they were too close. One thing that I found was that having the robot’s neck animated in the “too close” animation prevented me from programming the neck to turn to face the person when nodding. Even though the animation was inactive, having it in the animator caused the joint to be stuck. In order to solve this, I used the legacy animation system for the “too close” scenario, while using the animator component for everything else.

User Interface

The second component that I worked on was the UI using Unity’s UI features. This included a fully navigable menu with clickable breadcrumbs, sliders to customise speed and the field-of-view, and sound customization options that instantiate when the game is played.
